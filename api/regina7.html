Regina - Moving from 6.x to 7.x

This page relates only to C++ and Python programmers.

Over time, Regina makes changes and improvements to its C++ and Python interfaces. This page explains how you will need to update your C++ code and/or Python scripts to work with Regina 7.0.

Many of the old classes and routines have been kept around but marked as deprecated, so that your old code will still work.

All of this deprecated code will be removed in Regina 7.1.

If you have very old code, you can also read the separate pages on moving from Regina 4.x to 5.x and moving from Regina 5.x to 6.x.

Changes from Regina 6.x to 7.x

First, you should read the summary of the major behavioural changes, available here online.

There are some general rules that cover many of the changes:

  • TODO.
  • TODO: returning by value instead of by pointer
  • TODO: passing by const reference (or occasionally reference) instead of pointer
  • TODO: Using std::shared_ptr exclusively for packets
  • TODO: returning tuples instead of using reference arguments
  • Many functions now handle errors and/or unsolved scenarios differently. Instead of returning null or 0 or some other “trivial” value, these functions either return a std::optional or throw exceptions. TODO: discuss more.
  • Talk about throwing exceptions when preconditions break, where these preconditions were not previously being checked at all.
  • Talk about throwing exceptions instead of taking extra error output arguments (e.g., Rational::doubleApprox(), Integer(string)).
  • Talk about callbacks, including replacing void-pointer arguments (e.g., ModelLinkGraph::generateMinimalLinks(), FacetPairing::findAllPairings()), and more flexible interfaces such as Link::rewrite(). Sometimes output iterators are replaced with callbacks also, e.g., findAllIsomorphisms(). Final (null) calls are removed.
  • Sometimes routines that took optional output arguments now have separate versions, e.g., isoSig() vs isoSigDetail(), or volume() vs volumeWithPrecision().
  • Routines that take progress trackers no longer run in the background.
  • Routines Class::clone() are now replace by copy constructors.
  • For StandardTriangulation and its subclasses, the isXXX() routines are now all called recognise(), and the clone() routines are replaced with copy constructors. For example, StandardTriangulation::isStandardTriangulation() is now StandardTriangulation::recognise(), and AugTriSolidTorus::isAugTriSolidTorus() is now AugTriSolidTorus::recognise().
  • For C++ functions that take an output stream as argument, their Python bindings used to assume this stream to be standard output. Now these bindings have been removed entirely; in all cases there is an alternative available that returns a string instead. So, for example, instead of calling Link.writePACE(), Python users would now call print(Link.pace()).

Functions that now throw exceptions on error instead of returning null are:

  • Global functions makeMatchingEquations(), static member functions FacetPairing<dim>::fromTextRep(), GluingPermSearcher<dim>::readTaggedData(),
  • Input stream constructors for GluingPerms<dim>, GluingPermsSearcher<dim> and subclasses, the string constructors for GroupExpression, GroupPresentation;
  • All functions in HomMarkedAbelianGroup that used to return empty vectors on error;
  • Reconstruction reroutines: rehydrate(), insertRehydration(), fromSnapPea(), fromIsoSig(), fromPlantri(), magic string constructors, string constructors for GroupPresentation and GroupExpression, FacetPairing::fromTextRep()
  • Triangulation3::dehydrate(), snapPea(), recogniser(), recognizer() if not supported; likewise for Link::gauss(), orientedGauss(), dt() and knotSig().
  • SatRegion::createSFS()
  • HomMarkedAbelianGroup::...() that used to return empty vectors
  • ModelLinkGraph::flype()
  • Manifold::construct() and homology()
  • NormalSurface::boundaryIntersections()
  • Some member functions of SnapPeaTriangulation that used to return trivial output for null triangulations

Beyond these general rules, there are some additional changes and/or special cases. These are listed in the table below, along with suggested replacements that you can use instead. Any changes in the behaviour of these routines or classes are noted in the comments column.

The blue cells indicate code that is still present in Regina, and the red cells indicate code that has already been removed from the latest release.
